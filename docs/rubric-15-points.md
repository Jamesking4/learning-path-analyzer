# Критерии оценивания творческого задания по программированию (15 баллов)

## Общее описание

Задание оценивается по пяти категориям. Каждая категория содержит критерии, которые проверяются как автоматически (GitHub Actions), так и посредством анализа кода и документации.

---

## 1. ПОЛЕЗНОСТЬ ЗАДАЧИ (4 балла)

**Описание:** Задача должна быть актуальной, интересной и релевантной для специалиста в области ИИ в образовании, обработки данных или связанных направлений.

### Критерии оценивания:

**4 балла** — Отличная полезность
- Задача решает реальную проблему в выбранной предметной области
- Четко определена область применения (например: анализ образовательных данных, обработка текста, рекомендательные системы для обучения)
- Содержит элементы, которые можно использовать в реальных проектах
- Демонстрирует глубокое понимание проблемы

**3 балла** — Хорошая полезность
- Задача решает понятную проблему, хотя область применения может быть узкой
- Имеет практический потенциал
- Применима в образовательном контексте или для специалистов в смежных областях

**2 балла** — Средняя полезность
- Задача имеет образовательную ценность, но практическое применение ограничено
- Тема выбрана удачно, но реализация базовая
- Может быть полезна как учебный пример, но не более того

**1 балл** — Низкая полезность
- Задача хотя бы частично решена, но практическое применение минимально
- Тема слишком тривиальна или не выбрана

**0 баллов** — Задача отсутствует или не решена

## 2. ОФОРМЛЕНИЕ РЕПОЗИТОРИЯ (3 балла)

**Описание:** Качество структуры проекта, соответствие best practices GitHub/GitVerse.

### Критерии оценивания:

**3 балла** — Отличное оформление
- ✓ Присутствует `.gitignore` (соответствует типу проекта: Python, Node.js и т.д.)
- ✓ Наличие `requirements.txt` (или `setup.py`, `pyproject.toml`, `package.json`)
- ✓ Структура папок логична и интуитивна (например: `/src`, `/data`, `/tests`, `/docs`)
- ✓ Файлы распределены по смыслу (код, конфиги, документация разделены)
- ✓ Отсутствуют служебные файлы в репозитории (`.pyc`, `__pycache__`, `.env` и т.д.)

**2 балла** — Хорошее оформление
- ✓ Присутствует `.gitignore` и `requirements.txt`
- ✓ Структура папок логична, но может быть организована лучше
- ✓ Большинство файлов на месте, возможны небольшие недостатки (1-2 файла не в папке)
- ~ Могут быть незначительные проблемы с игнорированием файлов

**1 балл** — Удовлетворительное оформление
- ~ Присутствуют `.gitignore` и `requirements.txt`, но они неполные
- ~ Структура папок есть, но недостаточно логична
- ~ Нарушения в организации: смешивание кода и конфигов, отсутствие некоторых файлов

**0 баллов** — Плохое оформление
- ✗ Отсутствует `.gitignore` или `requirements.txt`
- ✗ Полный хаос в структуре репозитория
- ✗ Служебные файлы загружены в репо

## 3. РАБОТОСПОСОБНОСТЬ КОДА + АВТОПРОВЕРКИ (4 балла)

**Описание:** Функциональность кода и наличие CI/CD pipeline для контроля качества.

### Критерии оценивания:

**4 балла** — Код полностью работоспособен с хорошей CI/CD
- ✓ Все основные функции работают без ошибок
- ✓ GitHub Actions workflow (`.github/workflows/`) настроен корректно
- ✓ Запускаются проверки PEP 8 (flake8, black, pylint или аналог)
- ✓ Присутствуют unit tests или integration tests
- ✓ Badge в README показывает статус CI/CD (например, build passing)
- ✓ Pipeline запускается на каждый push/pull request

**3 балла** — Код работает, CI/CD частично реализован
- ✓ Основной функционал работает корректно
- ✓ GitHub Actions настроен и запускается
- ✓ Проверка кода присутствует (PEP 8 или тесты), но может быть неполной
- ~ Возможны незначительные ошибки, которые не влияют на основную логику
- ~ Тестов мало или они покрывают только часть кода

**2 балла** — Код работает, но CI/CD минимален
- ~ Основной функционал работает
- ~ GitHub Actions workflow есть, но проверки PEP 8 либо отсутствуют, либо очень базовые
- ~ Тесты отсутствуют или их мало
- ~ Ошибок нет, но качество кода не проверяется автоматически

**1 балл** — Код содержит ошибки или CI/CD отсутствует
- ~ Код запускается, но содержит ошибки в определенных случаях
- ~ GitHub Actions отсутствует или не работает
- ~ Нет автоматических проверок качества кода

**0 баллов** — Код не работает
- ✗ Ошибки при запуске
- ✗ GitHub Actions отсутствует

## 4. КАЧЕСТВО ДОКУМЕНТАЦИИ (README) (2 балла)

**Описание:** Полнота и ясность описания проекта в README.

### Критерии оценивания:

**2 балла** — Отличная документация
- ✓ Наличие всех ключевых разделов:
  - **Описание проекта** (что это, зачем нужно)
  - **Как установить** (installation)
  - **Как использовать** (usage с примерами)
  - **Структура проекта** (что где находится)
  - **Требования** (dependencies, Python version и т.д.)
  - **Примеры использования** (code examples с выводом)
- ✓ Текст ясный, без опечаток
- ✓ Примеры кода работают и актуальны

**1 балл** — Удовлетворительная документация
- ~ Присутствуют основные разделы, но какой-то из них неполный
- ~ Примеры есть, но не очень ясные
- ~ Документация описывает проект, но есть пробелы в описании использования

**0 баллов** — Плохая документация
- ✗ README отсутствует или содержит только название проекта
- ✗ Нет информации о том, как установить или использовать
- ✗ Примеры отсутствуют или не работают

## 5. КРЕАТИВНОЕ ИСПОЛЬЗОВАНИЕ CI/CD (2 балла)

**Описание:** Нестандартное и полезное использование GitHub Actions/GitVerse workflows.

### Критерии оценивания:

**2 балла** — Отличное творческое использование
- ✓ GitHub Actions используется не только для проверки кода, но и для:
  - **Автоматизации развертывания** (например, выкладывание результатов в ветку `gh-pages`)
  - **Запуска по расписанию** (scheduled workflow с `schedule` триггером)
  - **Обработки данных** (например, регулярное обновление datasets)
  - **Генерации отчетов** (автоматическое создание результатов и их коммит)
  - **Триггера на upload файлов** (использование `workflow_dispatch` или custom triggers)
  - **Уведомлений** (например, отправка результатов в Slack или почту)
- ✓ Workflow логичен, имеет практическую ценность
- ✓ Документирован в README (объяснено, что делает workflow)

**1 балл** — Хорошее использование CI/CD
- ~ GitHub Actions используется для чего-то большего, чем просто проверка PEP 8
- ~ Есть один из вышеописанных элементов (schedule, automation, reports и т.д.)
- ~ Workflow работает, но может быть более полезным

**0 баллов** — Стандартное использование или отсутствие
- ✗ GitHub Actions используется только для проверки синтаксиса
- ✗ GitHub Actions отсутствует

## Примеры творческого использования CI/CD:

### Пример 1: Автоматическое обновление данных
```yaml
on:
  schedule:
    - cron: '0 0 * * *'  # Ежедневно в полночь

jobs:
  update-data:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Update dataset
        run: python scripts/fetch_data.py
      - name: Commit changes
        run: |
          git config user.name "Data Bot"
          git add data/
          git commit -m "Update data: $(date)" || true
          git push
```

### Пример 2: Генерация отчетов при push
```yaml
on: [push]

jobs:
  generate-report:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Generate report
        run: python scripts/generate_report.py
      - name: Upload to artifacts
        uses: actions/upload-artifact@v3
        with:
          name: report
          path: reports/
```

### Пример 3: Интерактивный workflow
```yaml
on: workflow_dispatch

jobs:
  manual-task:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v3
      - name: Run custom analysis
        run: python scripts/analyze.py
```

## Рекомендации для студентов

1. **Начните с минимума:** требуемые файлы, базовая структура
2. **Добавьте тесты:** даже простые unit tests повысят оценку
3. **Будьте креативны с CI/CD:** это отличная возможность показать понимание DevOps практик
4. **Документируйте:** хороший README — это половина успеха
5. **Проверьте работоспособность:** убедитесь, что все примеры в README действительно работают
